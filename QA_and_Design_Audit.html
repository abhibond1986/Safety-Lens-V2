1Executive Summary

Safety Lens is a mature, feature-rich Flutter application delivering offline-first AI hazard scanning, near-miss reporting, WSA-13 analytics, and an admin control centre from a single shared codebase. The engineering is defensively coded against network failure and thoughtfully platform-aware. The two areas holding it back from an enterprise-grade rating are security debt in the authentication and API-key handling, and a visual identity that fought itself — a consumer-style violet/magenta glass theme layered over screens that already used a professional indigo/amber palette.

This audit delivers both a findings report and a set of applied changes: a unified industrial steel identity now cascades across every screen on both platforms, and several safe, high-value bug and security fixes have been made in place. Higher-risk security items are documented as recommendations rather than changed blindly, since they touch the live authentication contract with the Google Apps Script backend.

Testing note. The isolated Linux build environment was unavailable during this session, so the app could not be compiled or run live. All findings come from rigorous static review of the source; all applied edits were independently re-reviewed for syntax and regressions. A live flutter analyze and device smoke-test should be run before release to confirm.

2Expert Analysis

The review was conducted across two tracks in parallel — a services / backend pass (auth, sync, AI providers, storage, PDF, crypto) and a screens / widgets pass (state management, lifecycle, layout, accessibility). The picture that emerged is consistent: strong feature coverage and careful null-handling, undermined by security drift and design inconsistency accumulated across many rapid iterations.

What the codebase does well

  • Resilient offline architecture — a multi-provider AI fallback chain and a local knowledge-base fallback keep the app useful without a network.
  • Disciplined lifecycle in major flows — mounted checks after await, and disposal of long-lived controllers and timers, are consistently present in the primary tabs.
  • Platform awareness — conditional web/stub imports for PDF export and KB extraction show real attention to the web/Android split.
  • Clean separation of concerns — storage, sync, AI, i18n and PDF are cleanly modular.

Where the risk concentrates

  • Authentication — shipped default admin credentials, plaintext / hash-equals-password fallback login paths, and a hand-rolled SHA-256 that risks producing different hashes on web vs. mobile.
  • Secret handling — provider API keys are synced to the device and, for Gemini, placed in a request URL, contradicting the code's own "keys never leave the server" comments.
  • Auto-update — APKs are downloaded and installed with only a file-size check, no signature/checksum verification.
  • Design — clashing palettes, sub-legible 7–9px type, unlabeled icon-only controls, and heavy stacked blur in scrolling lists.

4Design Refresh Applied

The app previously mixed a deep purple-black glass theme (design tokens) with the professional indigo/cyan/amber palette used inside individual screens — so the hero, avatars and charts clashed with the content beneath them. The refresh unifies everything onto one deliberate industrial identity, chosen from the subject's own world: the graphite and steel-blue of a plant floor, with safety-amber as the recurring signature accent (the colour of plant signage and PPE).

Because the change was made at the design-token level — keeping every token's name identical — it ripples across every screen on both Web and Android with minimal regression risk, rather than requiring hundreds of scattered edits.

5Fixes Applied in This Pass

The following changes were made directly in the codebase and re-verified for syntax and regressions:

  • Unified design tokens in main.dart — new industrial palette, mesh & background gradients retuned to graphite/steel.
  • Removed clashing gradients in the home hero, both app-bar avatars, and three spots in the near-miss screen.
  • Stopped credential logging — the login hash and login diagnostics are now wrapped in assert() so they are stripped from release builds.
  • Fixed a controller memory leak — admin API-key fields are now disposed.
  • Fixed a potential crash — _userInitials no longer throws on names with trailing spaces.
  • Corrected compliance citation — generated PDFs and AI fallbacks now cite IS 14489:2018.
  • Removed a per-rebuild debug print loop in the AI result view.
  • Aligned the web splash background to the app's graphite base.
